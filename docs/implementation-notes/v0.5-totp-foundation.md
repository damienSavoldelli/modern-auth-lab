# v0.5.0 TOTP Foundation Implementation Notes

This note describes how the `v0.5.0 - TOTP Foundation` milestone should be approached.

The goal is to build TOTP understanding and core primitives before wiring TOTP into the full login flow.

## Learning Objective

At the end of this milestone, the project should make these concepts clear:

- what a TOTP secret is;
- how an authenticator app receives the secret;
- what `otpauth://` is used for;
- how a time-based code is generated;
- how the server verifies a submitted code;
- why enrollment must be confirmed before activation;
- why TOTP does not provide per-device revocation.

## Where The Work Should Happen

Expected backend areas:

- `src/Security/Totp/` for TOTP primitives such as secrets, URI generation, and verification.
- `tests/Security/Totp/` for focused unit tests.
- `docs/concepts/totp.md` for conceptual explanation updates.
- `docs/decisions/` for ADRs when persistence, encryption, enrollment state, or lifecycle decisions are introduced.

Expected later areas:

- `src/Infrastructure/Persistence/` when TOTP persistence is introduced.
- `src/Http/Controller/` when enrollment screens or actions are introduced.
- `src/Application/` when TOTP enrollment becomes a workflow instead of a primitive.

## Recommended Step Order

### Step 1: TOTP Concepts And Pure Logic

Implement only the pure logic:

- Base32-compatible secret representation.
- TOTP secret generation.
- HOTP/TOTP code calculation.
- TOTP verification with a small time window.
- `otpauth://` URI generation.

Do not add database persistence yet.

Stop after this step and validate understanding.

### Step 2: TOTP Persistence Model

Introduce persistence only after the pure TOTP logic is tested.

Model concepts to consider:

- pending secret;
- active secret;
- confirmed timestamp;
- last used time step;
- disabled or replaced state;
- revocation timestamp.

Stop after this step and validate the lifecycle model.

### Step 3: TOTP Enrollment

Introduce an authenticated enrollment flow:

- start enrollment;
- show QR code or provisioning URI;
- confirm with a valid code;
- activate TOTP only after confirmation.

Stop after this step and validate the user/security flow.

## Explicit Non-Goals For v0.5.0

- Do not require TOTP during login yet.
- Do not replace the full session login flow yet.
- Do not add Passkeys/WebAuthn.
- Do not implement recovery flows.
- Do not implement trusted devices.
- Do not implement SMS or email OTP.
- Do not build a complex frontend experience.

## Documentation Rule

Each implementation step must update documentation when it introduces a concept that matters for security or future article writing.

The explanation should include:

- the problem being solved;
- the security risk;
- the chosen implementation;
- the tradeoffs;
- what remains intentionally deferred.

After each major block, stop and ask whether the explanation is clear before continuing.
